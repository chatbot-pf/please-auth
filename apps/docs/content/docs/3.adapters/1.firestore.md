# Firestore Adapter

`@please-auth/firestore` provides a Cloud Firestore database adapter for Better Auth with full CRUD, transactions, snake_case field mapping, and custom collection names.

## Setup

```ts
import { betterAuth } from "better-auth";
import { firestoreAdapter } from "@please-auth/firestore";
import { getFirestore } from "firebase-admin/firestore";

const auth = betterAuth({
  database: firestoreAdapter({ db: getFirestore() }),
});
```

## Options

### `db`
- **Type:** `Firestore`
- **Required:** Optional in the TypeScript type, but required at runtime — omitting it will cause an error unless a default Firebase app is already initialized.

The Firestore instance from `firebase-admin/firestore`.

### `namingStrategy`
- **Type:** `"default" | "snake_case"`
- **Default:** `"default"`

Field naming strategy:
- `"default"` — uses camelCase field names as-is from Better Auth.
- `"snake_case"` — maps camelCase fields to snake_case in Firestore (e.g., `userId` → `user_id`).

```ts
firestoreAdapter({
  db: getFirestore(),
  namingStrategy: "snake_case",
});
```

### `collections`
- **Type:** `Partial<CollectionNames>`
- **Default:** `{ users: "users", sessions: "sessions", accounts: "accounts", verifications: "verifications" }`

Override default Firestore collection names:

```ts
firestoreAdapter({
  db: getFirestore(),
  collections: {
    users: "app_users",
    sessions: "auth_sessions",
  },
});
```

### `debugLogs`
- **Type:** `boolean`
- **Default:** `false`

Enable debug logging for adapter operations.

## `initFirestore` Helper

A convenience function to initialize or reuse a Firestore instance. Safe for serverless environments — it reuses the existing Firebase app if one is already initialized.

```ts
import { initFirestore } from "@please-auth/firestore";

const db = initFirestore({ projectId: "my-project" });
```

Use it with the adapter:

```ts
import { betterAuth } from "better-auth";
import { firestoreAdapter, initFirestore } from "@please-auth/firestore";

const auth = betterAuth({
  database: firestoreAdapter({
    db: initFirestore({ projectId: "my-project" }),
  }),
});
```

## Field Mapping

When using `namingStrategy: "snake_case"`, the following field mappings are applied automatically:

| Better Auth (camelCase) | Firestore (snake_case) |
|-------------------------|------------------------|
| `userId` | `user_id` |
| `sessionToken` | `session_token` |
| `providerAccountId` | `provider_account_id` |
| `emailVerified` | `email_verified` |
| `accessToken` | `access_token` |
| `refreshToken` | `refresh_token` |
| `idToken` | `id_token` |
| `accessTokenExpiresAt` | `access_token_expires_at` |
| `refreshTokenExpiresAt` | `refresh_token_expires_at` |
| `expiresAt` | `expires_at` |
| `createdAt` | `created_at` |
| `updatedAt` | `updated_at` |

## Features

### Transaction Support

The adapter fully supports Firestore transactions for atomic operations:

```ts
// Better Auth handles transactions internally.
// The adapter wraps operations in db.runTransaction()
// ensuring consistency across multi-document updates.
```

### ID-based Fast Paths

The adapter optimizes queries that filter by document ID — it uses direct document lookups (`doc.get()`) instead of queries, significantly reducing read costs.

### OR Query Support

Complex `OR` queries are split into parallel Firestore queries and merged client-side for broader compatibility, as native Firestore `OR` filters have constraints on compound queries.

### Batch Operations

`updateMany` and `deleteMany` use Firestore batched writes (up to 500 operations per batch) for efficient bulk operations.

## Supported Capabilities

| Capability | Supported |
|------------|-----------|
| JSON fields | Yes |
| Date fields | Yes |
| Boolean fields | Yes |
| Numeric IDs | No (Firestore uses string document IDs) |
| Transactions | Yes |
| Batch writes | Yes |
